design architecture of CYCLE01 is fundamentally and irrevocably driven by the extensive use of
Pydantic domain models. This approach ensures exceptionally strict data typing and uncompromising
validation throughout the entire ETL data pipeline. This rigorous Domain-Driven approach
categorically prevents the dangerous proliferation of implicit assumptions regarding data structures
and guarantees, with absolute certainty, that the system always operates on known, perfectly valid
data. All these core definitions are centrally and securely housed within the `src/domain/models.py`
file.  The primary and most critical data entity is the `RawQuote` model. This specific Pydantic
class precisely defines the exact expected structure of the raw data precisely as it is received
from the external J-Quants API. It explicitly specifies the exact fields that are absolutely
required—such as the specific `date`, the `open` price, the `high` price, the `low` price, the
`close` price, and the trading `volume`—and it mercilessly enforces their precise data types (for
example, ensuring unequivocally that all prices are represented as floating-point numbers and that
the volume is always an integer). Crucially, this domain model implements sophisticated internal
validation rules to guarantee absolute data integrity. For example, it utilizes custom field
validators to enforce the inviolable invariant that the `high` price must always be greater than or
equal to both the `low` price, the `open` price, and the `close` price. If the external API happens
to return anomalous, corrupted, or logically impossible data that violates these fundamental rules,
Pydantic will instantly raise a catastrophic validation error, effectively halting the process and
absolutely preventing the corrupt data from ever entering the internal transformation pipeline. This
model acts as the ultimate, strict, and uncompromising contract between the unpredictable external
world and the highly controlled internal system.  Building directly upon the foundation of the raw
data, the `ProcessedQuote` model defines the exact structural blueprint of the fully enriched data,
precisely ready for complex statistical analysis. This model is carefully designed; it inherits
directly from `RawQuote` (or strategically includes it as a core nested component) and explicitly
adds entirely new fields for all the mathematically calculated features. These explicitly required
fields include the `day_of_week` (which is strictly and mathematically constrained to be an integer
precisely between 1 and 5), the `is_month_start` flag (strictly a boolean), the `is_month_end` flag
(strictly a boolean), the `daily_return` metric (strictly a float), the `intraday_return` metric
(strictly a float), and the `overnight_return` metric (strictly a float). By explicitly and
rigorously defining this schema, we systematically establish an incredibly clear, unbreakable
contract strictly between the processing layer (acting as the producer) and the storage/analysis
layers (acting as the critical consumers). The consumers can rely with absolute mathematical
certainty on the fact that any data they receive has definitively passed this extraordinarily strict
validation process.  The fundamental configuration of the entire system is also strictly governed by
Pydantic, specifically utilizing the `AppSettings` class meticulously defined in
`src/core/config.py`. This model directly inherits from `pydantic_settings.BaseSettings` and
explicitly and uncompromisingly declares all the absolutely required environment variables,
specifically mandating the presence of the `JQUANTS_REFRESH_TOKEN`. This architectural decision
ensures that the application robustly fails incredibly fast, during the very initial startup phase,
if the necessary cryptographic credentials are not provided. This proactively prevents deeply
confusing and difficult-to-debug runtime errors from occurring much later in the execution cycle.
The design architecture, therefore, deliberately and systematically utilizes Pydantic not merely for
simple data serialization, but as an incredibly comprehensive, robust, and unbreakable validation
and contract-enforcement mechanism operating flawlessly across all critical architectural
boundaries. This guarantees complete backward compatibility, meticulous versioning control, and
ultimate, unwavering system stability.  ## Implementation Approach The detailed implementation of
CYCLE01 will proceed highly methodically, beginning immediately with the core configuration and

`intraday_return` and the `overnight_return` utilizing highly optimized column arithmetic

decision provides a perfectly clean, highly abstract interface specifically designed for the
