# CYCLE 02 SPECIFICATION: Backtesting & Stats

## 1. Summary
The objective of Cycle 02 is to build the Statistical Testing and Backtesting modules. Utilizing the Parquet datasets generated by the Data Pipeline in Cycle 01, this cycle will implement robust statistical analysis using `statsmodels` to determine if specific days of the week exhibit statistically significant abnormal returns. Furthermore, it will integrate `vectorbt` to transform the Polars data frames into an actionable format for algorithmic trading backtesting. The backtest will simulate buying and selling on specific days (e.g., Buy Monday, Sell Friday) while rigorously applying transaction costs (fees/slippage) to generate realistic Sharpe ratios and win rates. Finally, a complete E2E integration test will run through both Cycle 01 and Cycle 02 functionalities.

## 4. Design Architecture
The system is fully designed by Pydantic-based schema.

### Domain Concepts
- **`domain/analysis.py`**: Defines new schemas for analysis outputs.
  - `StatResult`: A Pydantic model representing the result of a statistical test (e.g., target day of week, p-value, t-statistic, whether it's significant).
  - `BacktestMetrics`: A Pydantic model representing the final output of a VectorBT run (e.g., Total Return, Sharpe Ratio, Win Rate, Max Drawdown).

### Key Invariants, Constraints, and Validation Rules
- The Backtesting engine must validate that the provided dataframe is not empty and that the `Close` and `Open` columns contain valid floats. Empty subsets must raise a `ValueError`.
- `statsmodels` tests must use heteroskedasticity-robust standard errors (e.g., HC1/HC3) when calculating OLS given the nature of financial returns.
- Pydantic models must enforce strict validation (`extra="forbid"`).

### Expected Consumers
- The primary consumer for `analysis/stats.py` and `analysis/backtest.py` will be the End-to-End tutorial notebooks (`UAT_AND_TUTORIAL.py`).

## 5. Implementation Approach

1.  **Domain Output Modeling**:
    -   Create `src/domain/analysis.py` and define the `StatResult` and `BacktestMetrics` Pydantic models.
2.  **Statistical Testing (`statsmodels`)**:
    -   Implement `src/analysis/stats.py`.
    -   Create a function that takes a DuckDB-polars relation or standard Polars DataFrame as input.
    -   Filter the dataframe to isolate returns for a specific target `day_of_week`.
    -   Run an OLS regression or T-Test using `statsmodels` comparing the target day's returns against the population or another day, checking if the p-value is less than 0.05. Return a `StatResult`.
3.  **Backtesting Engine (`vectorbt`)**:
    -   Implement `src/analysis/backtest.py`.
    -   Create a function that accepts the Polars dataframe. Convert the relevant columns (`Close`, date index) into pandas Series/DataFrames as required by VectorBT natively.
    -   Generate boolean arrays for entry signals (e.g., buy on Monday open) and exit signals (e.g., sell on Friday close).
    -   Execute `vectorbt.Portfolio.from_signals` supplying the prices, entries, exits, and a specific fees parameter (e.g., 0.001 for 0.1%).
    -   Extract the performance metrics (Sharpe, Win Rate) and return them typed as `BacktestMetrics`.
4.  **Integration**:
    -   Update any necessary `__init__.py` files to expose these modules cleanly.

## 6. Test Strategy

### Unit Testing Approach (Min 300 words)
The unit testing for the analysis module will heavily rely on synthetic data generation to ensure stability and avoid flakiness.
For `analysis/stats.py`, the test will generate a Polars dataframe using a seeded `numpy.random.normal` distribution where Mondays are artificially injected with a constant high positive mean, and Fridays with a negative mean. The test will run the statistical function and assert that the returned `StatResult` correctly identifies the Monday anomaly as statistically significant (p-value < 0.05) and returns a specific, expected T-statistic. A counter-test with completely random uniform data will assert that the p-value is > 0.05.
For `analysis/backtest.py`, tests will utilize a predictable, linear price series (e.g., `[100, 101, 102, 103, 104...]`). By feeding a deterministic "buy on index 0, sell on index 3" signal, the test can manually calculate what the total return, minus fees, should be. It will assert that the `BacktestMetrics` output by VectorBT matches this manual calculation exactly. This ensures that the engine is correctly interpreting the `from_signals` arguments, particularly fees and slippage.

### Integration Testing Approach (Min 300 words)
Integration tests for Cycle 02 will bridge the gap between Cycle 01 storage outputs and Cycle 02 analysis engines. Utilizing `test_analysis.py`, a fixture will execute the Cycle 01 `storage/parquet_db.py` logic to write a temporary `.parquet` file to disk utilizing `tmp_path`. This file will contain synthetic realistic quote data.
The integration test will then initialize the DuckDB reader, fetch the Polars dataframe, and pass it directly into the `backtest.py` and `stats.py` methods. It will verify that the entire chain—from disk storage, through schema validation, into `statsmodels`, and out to `vectorbt`—executes without type errors, missing column errors, or memory faults. This tests the compatibility of the Polars -> Pandas conversion required by VectorBT natively.
Finally, a holistic pseudo-E2E test will be orchestrated without network access. It will utilize a mocked payload from the ingestion layer, run the Polars transformation, save it to disk, load it via DuckDB, run the backtest, and output the final Pydantic metric model, successfully mimicking the entire production workflow while maintaining sandbox resilience.
