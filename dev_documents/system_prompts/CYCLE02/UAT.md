# CYCLE02: User Acceptance Testing (UAT) Plan

## Test Scenarios

### Scenario ID: UAT-02-01
**Priority:** High
**Title:** Calendar Anomaly Statistical Validation
**Description:** This scenario enables the quantitative researcher to confirm the presence or absence of calendar anomalies using rigorous statistical proofs within the interactive notebook environment. After executing the data pipeline cells from Cycle 01, the user will trigger the Statistical Analysis cell in the Marimo notebook. The user interface will allow the researcher to select a specific target day (e.g., Monday). The underlying engine will query the stored Parquet data via DuckDB, separate Monday returns from non-Monday returns, and execute a statistical T-test. The UI must then render the resulting `StatResult` Pydantic model in a clear, readable format. Crucially, the UI should visually highlight the P-value and the `is_significant` boolean flag (e.g., green for significant, red for insignificant), allowing the researcher to immediately understand whether the "Monday Effect" is statistically proven within the ingested dataset.

### Scenario ID: UAT-02-02
**Priority:** High
**Title:** Algorithmic Backtest Execution and Performance Evaluation
**Description:** This scenario verifies the core financial utility of the system: simulating a trading strategy based on the discovered anomalies. Within the Marimo notebook, the user will execute the Advanced Algorithmic Backtesting cell. This cell will configure a predefined trading signal, such as entering a long position on the Monday open and exiting on the Friday close. The notebook will pass this signal, along with the historical price data, into the `vectorbt` backtesting engine. The system must account for realistic trading frictions, specifically applying a 0.1% transaction fee. Upon completion, the notebook will render the `BacktestMetrics` Pydantic model. The user will verify that the Sharpe Ratio, Win Rate, and Total Return are displayed correctly. Furthermore, the notebook should output a visual equity curve generated by `vectorbt`, allowing the analyst to visually inspect the simulated portfolio growth and drawdowns over the 12-week period.

## Behavior Definitions

```gherkin
Feature: Statistical Validation of Calendar Anomalies
  As a quantitative researcher
  I want to perform statistical tests on the transformed stock data
  So that I can mathematically prove whether anomalies like the Monday effect exist.

  Scenario: Testing for the Monday Effect
    Given a DuckDB connection to the transformed historical stock data
    When the user requests a statistical comparison of Monday returns against the rest of the week
    Then the system should execute an independent T-test using statsmodels
    And output a structured `StatResult` object containing the T-statistic and P-value
    And explicitly flag the result as significant if the P-value is less than 0.05.

Feature: Algorithmic Backtesting Simulation
  As a quantitative analyst
  I want to simulate trading strategies based on day-of-week signals
  So that I can evaluate the real-world profitability of the anomalies, accounting for costs.

  Scenario: Executing a week-long hold strategy with fees
    Given a Polars DataFrame containing daily historical quotes and calculated features
    When the system generates a signal to "Buy Monday Open" and "Sell Friday Close"
    And executes the backtest through the vectorbt engine
    And applies a fixed transaction fee of 0.1% per trade
    Then the system must successfully complete the simulation without errors
    And output a `BacktestMetrics` object containing the `sharpe_ratio`, `win_rate`, and `total_return`
    And the total return must mathematically reflect the deduction of the transaction fees.
```
